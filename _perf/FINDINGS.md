[HIGH] FIRESTORE | UserProfileView.swift:938–946 | `fixFollowerCounts()` fetches the entire `follows` collection twice with no `.limit()` — one read for all followers, one for all following. A power user with 10k+ followers triggers a 10k-document read on every profile open that hits the corrupt-count code path. | Replace with Firestore aggregation queries (`count()`) from SDK 10.x, or move this repair logic behind a dedicated Cloud Function triggered offline. | Eliminates O(N) reads; shrinks a potential 10k-doc round-trip to a single count response.

[HIGH] FIRESTORE | FirebaseMessagingService.swift:876–879 | `getOrCreateConversation()` fetches all conversations where `participantIds arrayContains currentUserId` with no `.limit()`. A user with hundreds of conversations triggers a full collection scan on every new DM open. Same pattern repeated at line 2934 (`deleteConversationsWithUser`) and line 2961 (`archiveConversationsWithUser`). | Add `.limit(to: 200)` on the lookup query; move bulk delete/archive operations to a Cloud Function that pages server-side. | Cuts read cost from O(all conversations) to O(1 result) on the hot path.

[HIGH] FIRESTORE | MessagingImplementation.swift:251–254 | `getOrCreateConversation()` repeats an unbounded `conversations` scan (no `.limit()`). Duplicate pattern to FirebaseMessagingService — two code paths for the same operation means double the cost and diverging behaviour. | Consolidate to a single canonicalized service method with `.limit(to: 1)` querying the specific user pair only. | Eliminates O(N) conversation scan and the code-duplication maintenance hazard.

[HIGH] FIRESTORE | BereanChatView.swift:940 | `deleteConversation()` fetches the entire `messages` subcollection with no `.limit()` before deleting: `convRef.collection("messages").getDocuments()`. A long Berean conversation could hold thousands of messages, all fetched to the client before deletion. | Move deletion to a Cloud Function that paginates deletes server-side in batches of 500 (Firestore batch limit). | Eliminates potentially huge message reads on device; prevents OOM on large conversations.

[HIGH] FIRESTORE | AntiHarassmentEngine.swift:192–196, 309–312, 625–629, 692–697, 951–955 | Five separate unbounded reads: `getEnforcementHistory()` (line 196), targeting check (line 312), `getUserAppeals()` (line 629), directed harassment patterns (line 697), and `getActiveRestrictions()` (line 955) — all query collections with no `.limit()`. An adversarial user accumulating records could generate unbounded reads. | Add `.limit(to: 100)` on all enforcement history queries (thresholds are single-digit counts; 100 is ample); `.limit(to: 20)` on appeals; `.limit(to: 50)` on restrictions. | Caps read size for safety collections; closes an abuse vector.

[MEDIUM] FIRESTORE | ChannelService.swift:103–117 | `listenCommunalMessages()` and `listenSacredMessages()` attach snapshot listeners to message subcollections with no `.limit()`. As a channel grows, every snapshot fires with the full conversation history. | Add `.order(by: "createdAt", descending: true).limit(to: 100)` to both listener methods, consistent with how FirebaseMessagingService handles DM lists. | Prevents O(N-messages) snapshot payload on every new message in active channels.

[MEDIUM] FIRESTORE | BereanConversationService.swift:246–252 | `listenToMessages()` attaches a snapshot listener with no `.limit()`. Berean conversations accumulate indefinitely. | Add `.order(by: "createdAt", descending: true).limit(to: 50)` matching the pattern already used in `BereanChatViewModel` (line 69). | Aligns the two Berean message listeners; eliminates unbounded first-load snapshot.

[MEDIUM] FIRESTORE | UserProfileView.swift:820–826 | `setupRealtimePostsListener()` listens to all posts for a user with no `.limit()`. A prolific creator with 500+ posts causes the initial snapshot to return all posts to the device. | Add `.limit(to: 50)` and use the existing cursor-based `loadMorePosts()` for older content. | Reduces initial listener payload from hundreds of docs to a bounded 50.

[MEDIUM] FIRESTORE | FirebasePostService.swift:2573–2596 | `enrichPostsWithProfileImages()` fires one `db.collection("users").document(authorId).getDocument()` per unique author inside a serial `for` loop. [HIGH] NETWORK | ClaudeService.swift:109–124 + ClaudeAPIService.swift:150–170 | Berean streaming is fully simulated: `ClaudeService.sendMessage()` awaits the COMPLETE response from `bereanChatProxy` (a blocking Cloud Function call) before emitting the first character via typewriter playback at 15 ms/char. The user sees a blank/thinking state for the full LLM round-trip (~1–4 s). For a 400-word response (~2400 chars) typewriter playback alone takes ~36 s of continuous character emission. | Option A: Wire real SSE via a Cloud Run proxy that streams chunked HTTP from Anthropic directly. Option B: Keep CF approach but adopt sentence-level chunked emission at ~60–80 ms/sentence so full response is visible in 3–5 s, not 36 s. At minimum add a progress bar during the blocking CF wait. | First-token perceived latency drops from full round-trip to near-zero (A) or visible progress during wait (B). Playback duration cut from 36 s to 5 s.

<!-- ── CLOUD FUNCTION TIMEOUTS ─────────────────────────────────────────────── -->

[HIGH] NETWORK | ClaudeService.swift:225–233, ClaudeAPIService.swift:115, and 726 additional CF call sites | 728+ `httpsCallable("...").call(data)` calls across the app use no `timeoutInterval`. Firebase SDK default timeout is 70 seconds. On poor signal, the UI hangs for over a minute with no recoverable error. Only `FeedAPIService.swift` (3 s) and `ModelRoutingEngine.swift` (30 s) set custom timeouts. | Create a shared helper `func callWithTimeout(_ name: String, data: Any, timeout: TimeInterval = 15)` that wraps the callable with a racing `Task.sleep` sentinel (matching the pattern already used in `BereanChatViewModel` at line 587–594 for the stream). Audit all CF call sites and replace raw `.call()` with the helper. | Reduces worst-case hang from 70 s to 15 s; recoverable error state shown to user after timeout.

<!-- ── SERIAL AWAIT CHAINS ──────────────────────────────────────────────────── -->

[HIGH] NETWORK | ProfileView.swift:958–986 — `refreshProfile()` | Four independent data sources fetched serially: `postService.fetchUserPosts` → `savedPostsService.fetchSavedPosts` → `commentsService.fetchUserCommentInteractions` → `repostsService.fetchUserReposts`. Each is an independent network call. Total latency = sum of all four. The correct `async let` pattern already exists in `loadProfileData()` at line 1199 but was NOT applied to `refreshProfile()`. | Replace serial awaits with `async let postsTask / savedTask / repliesTask / repostsTask` + tuple await matching `loadProfileData()` at line 1199. | Pull-to-refresh latency drops from sum(P95 of 4 calls) to max(P95 of 4 calls) — approximately 3× faster (typical: ~2400 ms → ~800 ms).

[MEDIUM] NETWORK | UserProfileView.swift — `loadProfileData()` (~line 1155) | For external profiles, `checkFollowStatus()` and `checkPrivacyStatus()` run sequentially (note at line 1152 documents intentional serialization due to `swift_task_dealloc` risk). After the parallel posts/reposts fetch, `PrivacyAccessControl.shared.relationship(to:)` is called a THIRD time at line 1158, despite `checkPrivacyStatus()` already computing the relationship internally. | Audit whether line 1158 is redundant. If `checkPrivacyStatus()` already sets `viewerRelationship`, remove the duplicate call at line 1158. | Saves one Firestore read per external profile open.

<!-- ── STREAMING LIFECYCLE / CANCEL PATH ──────────────────────────────────── -->

[MEDIUM] NETWORK | ClaudeService.swift:120–124 | Typewriter playback loop uses `try? await Task.sleep(...)` which silently swallows `CancellationError`. If the user dismisses `BereanChatView` during typewriter playback, the loop continues running in the background for up to 36 s for a long response, consuming CPU and preventing `isProcessing` from clearing. The outer `catch is CancellationError` at line 128 only fires after the loop exits. | Change `try? await Task.sleep(nanoseconds: ...)` to `try await Task.sleep(nanoseconds: ...)` so cancellation propagates immediately out of the character loop. | Eliminates zombie typewriter task; CPU-free immediately on dismiss; `isProcessing` clears correctly.

<!-- ── SEARCH DEBOUNCE ──────────────────────────────────────────────────────── -->

[MEDIUM] NETWORK | SearchViewComponents_New.swift:134–140 | `onChange(of: searchText)` cancels the previous task and immediately creates a new one calling `performSearch(query:)`, which then internally does its own 300 ms `Task.sleep` debounce. The double-layer (cancel-at-change-site + sleep-inside) is redundant: the cancel pattern at the `onChange` site already provides debounce semantics; the internal 300 ms sleep just adds latency to every query that fires after a pause. | Remove the internal 300 ms sleep from `performSearch()` and rely on the external cancel-and-recreate, OR keep the sleep and remove the external task cancel. Do not double-layer. | Cuts search response latency by 300 ms for users who type then pause.

[HIGH] CREATE-POST | CreatePostView.swift:4105–4305 | Five sequential async safety await calls on the hot publish path block UI for 1–4+ seconds before any Firestore write begins. Order: (1) ModerationIngestService.check (local+network), (2) AmenBotDefenseService.evaluateBeforeAction (CF), (3) AmenContentPreflightService.runFinalPreflight (CF), (4) ThinkFirstGuardrailsService.checkContent (local+CF chain), (5) ThinkFirstServerValidator.validate (CF round-trip). All five run serially inside the Task started at line 4106. User sees "isPublishing = true" spinner but no optimistic content for the entire duration. | Parallelize independent guards using async let or withTaskGroup; run the two local checks (ModerationIngestService + ThinkFirstGuardrailsService) in parallel with AmenBotDefense; run ThinkFirstServerValidator concurrently with the local ThinkFirst check and take the stricter result. | ~600–1400 ms off the visible wait before Firestore write fires.

[HIGH] CREATE-POST | CreatePostView.swift:4808–5341 | No optimistic insert before Firestore write. After all safety gates pass, the Firestore write happens at line 5377 with visibility="under_review". The real-time listener is explicitly NOT fired (comment at line 5537: "Do NOT send .newPostCreated with isOptimistic=true"). The feed only updates after the server-side serverSidePostModeration CF promotes visibility to "everyone". For a text-only post on a fast connection this is still 500–2000 ms of blank-feed time after the user sees the "being reviewed" notice. | Either: (a) Insert a local optimistic post immediately after the safety gates pass (before the Firestore write) and flag it with a "pending moderation" shimmer banner, then replace with the server-confirmed post when the listener fires; or (b) show the compose sheet "Posting…" pill and dismiss immediately, scrolling to a sticky "in-review" placeholder in the feed that resolves via listener. | Feed feels immediately populated; matches Threads/Instagram UX.

[HIGH] CREATE-POST | CreatePostView.swift:5305–5312 | Idempotency pre-check reads the Firestore document before every post attempt (getDocument at line 5307–5311). This adds one synchronous network RTT (~100–300 ms) to every publish path even when no duplicate exists. The pre-check cannot succeed optimistically — it is a cold read with no offline cache. | Remove the client-side pre-check. The server idempotency key (idempotencyKey field in postData + backend duplicate detection) is the authoritative gate. Client-side duplicate prevention is already handled by the inFlightPostId guard at line 3920. | Eliminates one blocking Firestore read (~200 ms) per post attempt.

[MEDIUM] CREATE-POST | CreatePostView.swift:4921–4925 | User profile (profileImageURL, username) is fetched from Firestore mid-publish inside the Task (userDataTask at line 4920). This is a cold read that blocks post creation until the user document round-trip completes (~100–400 ms), even though UserDefaults already caches `currentUserProfileImageURL` (line 44) and username from prior sessions. | Read profileImageURL and username from UserDefaults/AppStorage cache first; only fall back to Firestore fetch if the cached value is stale or empty. This is fire-and-forget for the UI — the Firestore post document can write the cached value and be patched asynchronously by the normalizePostAuthorOnCreate CF. | Removes one blocking Firestore read from the critical publish path.

[MEDIUM] CREATE-POST | CreatePostView.swift:4027–4032 | SupportDetectionService.shouldPresentSupportDraftGate is called synchronously on the main thread in publishPost() at line 4027, before the background Task is spun up. If SupportDetectionService blocks (pattern matching across long strings), this stalls the main thread and delays the keyboard dismissal. | Move the support draft gate check into the background Task alongside the other safety checks. | Keeps main thread free during validation.

[LOW] CREATE-POST | CreatePostView.swift:335–344 | CreatePostView initializer (init) sets @State initial values for selectedCategory, commentPermission, and postVisibility synchronously — this is fine. However, the view carries 60+ @State variables and three @StateObject properties (linkController, cameraCoordinator, scriptureDetectService) allocated at init time. The compose sheet is noticeably heavy on first presentation. | Audit which @StateObject properties can be deferred to `.task` or lazy-initialized (e.g., ScriptureAutoDetectService does not need to be live until text appears; ComposerLinkPreviewController does not need to be live until a URL is detected). | Faster sheet presentation, lower first-frame RAM spike.

[LOW] CREATE-POST | CreatePostView.swift:6218–6231 | compressImage is marked nonisolated but calls UIImage(data:) and .jpegData() — UIKit operations that must remain on the correct thread. It is called inside Task.detached (line 5955) which correctly offloads to a background thread. However the while-loop compression at lines 6225–6227 can run up to 8 iterations at 1 MB target size. ## SEARCH LAYER

[HIGH] SEARCH | DiscoverSearchComponents.swift:189–211 | scheduleSearch debounce is 350 ms. This is within acceptable range and correctly cancels the previous debounceTask before scheduling a new one. However, executeSearch at line 214 cancels the previous searchTask (line 217) but does NOT await it — it immediately starts a new 8-collection withTaskGroup. If network latency is high, multiple searchTasks can run concurrently for a short window until the previous task's task group cancellation propagates. | After searchTask?.cancel(), add a guarded check: ensure isLoading is set to false and results are cleared before the new task sets isLoading = true, so the UI never shows stale results from a racing task. Additionally consider a hard serial gate (a single actor-serialized search slot) to fully prevent overlap. | Eliminates rare result-overwrite race where old results briefly appear before new ones.

[HIGH] SEARCH | DiscoveryService.swift:95–101 | DiscoveryService.updateSearch debounces typeahead at 200 ms, below the 300 ms minimum recommended. At 200 ms a fast typist (5 keys/sec) still fires 3–4 Algolia typeahead calls per second instead of 1–2. Combined with AlgoliaSearchService.getUserSuggestions, which makes a live AlgoliaSearch SDK RPC per call, this results in excess Algolia usage cost and potential quota throttling. | Raise typeahead debounce from 200 ms to 300–350 ms to match UniversalSearchViewModel.scheduleSearch (350 ms). | Reduces Algolia typeahead RPCs by ~30%, lowers cost and network contention.

[MEDIUM] SEARCH | AlgoliaSearchService.swift:53–55 | Result cache is keyed on full query string (suggest:<query_lowercased>) with a 5-minute TTL. The cache covers getUserSuggestions and userSearchCache but does NOT cover searchPosts or searchChurches. A user searching "grace testimony" then navigating away and returning re-fires the full 8-collection Firestore search (UniversalSearchViewModel.executeSearch) from scratch — no cache. | Extend the cache to cover UniversalSearchResults keyed on (scope, query). Use the same 300-second TTL. | Eliminates redundant 8-collection Firestore fan-out for repeated queries within 5 minutes.

[MEDIUM] SEARCH | SemanticSearchService.swift:47–54 | SemanticSearchService.search performs a Firestore collection scan of 100 posts from the last 90 days on every call. There is no debounce: callers invoke search(query:) directly. A search triggered from the Berean search UI (BereanSemanticSearch.swift) that is not debounced at the call site would re-run this full Firestore scan on every query change. | Wrap SemanticSearchService.search in a 400 ms debounce at the call site (via a Task-cancel pattern) wherever it is triggered by text field onChange. Additionally cache the 100-post snapshot for 60 seconds so multiple sub-second queries reuse the same data. | Avoids repeated full Firestore scans on rapid typing.

[MEDIUM] SEARCH | AlgoliaSearchService.swift:26–87 | AlgoliaSearchService initializes and configures the SearchClient eagerly at app boot (via the shared singleton's private init at line 57). If AlgoliaConfig credentials are not yet available (e.g., the app is fetching them from Remote Config after first launch), setupAlgoliaClient logs a warning and sets client = nil — all subsequent searches silently fail until the app is restarted. | Initialize AlgoliaSearchService lazily on first search call, not at singleton creation. Add a public reinitialize() method that callers invoke after Remote Config loads the real key. | Prevents silent Algolia failures on first-launch / RC-key scenarios.

[LOW] SEARCH | DiscoverSearchComponents.swift:243–280 | The 8-collection parallel Firestore search (withTaskGroup) issues all 8 queries simultaneously. The results struct is rebuilt atomically only after ALL 8 complete (line 269). If one slow query (e.g., books/events with sparse Firestore data) takes 2+ seconds, the entire results object is withheld from the UI. | Return partial results progressively: publish the results struct after each collection completes, using a @Published var for each section separately, so people and posts (fastest) appear immediately while churches/books load. | Search results start appearing in ~300 ms instead of waiting for the slowest query.
