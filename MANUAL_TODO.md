## 🚨 BLOCKERS (must resolve before App Store submission)

- [ ] **GIVING / IAP decision** — Decide merchant-of-record + nonprofit status before building more of Giving. In-app Stripe collection by a non-Apple-approved nonprofit = rejection under Guideline 3.2.1. Choose:
  - **(A) Approved-nonprofit path**: apply via Apple's Nonprofit Program, require Apple Pay, ensure every listed church also has Apple nonprofit approval; OR
  - **(B) Out-of-app path**: move Giving to SFSafariViewController / external browser, AMEN never collects funds in-app.
  - **WHY**: Consistent App Store rejection pattern for this exact stack (native + Stripe + non-approved nonprofit). Cannot be coded around.
  - **WHERE**: Business/legal decision first; then App Store Connect nonprofit enrollment; then architecture change (HYBRID: agent re-architects the code once decided).

- [ ] **Sign in with Apple** — If any social login (Google) is offered, Apple requires Sign in with Apple as an equal option (Guideline 4.8).
  - **WHY**: Mandatory when any third-party login is present; rejection risk.
  - **WHERE**: Apple Developer portal → enable "Sign in with Apple" capability → add to App ID → Xcode Signing & Capabilities. (HYBRID: agent adds the SwiftUI button + AuthenticationServices code once capability is enabled.)

- [ ] **Age-rating questionnaire** — New Apple 13+/16+/18+ tiers effective July 2025. Must complete updated questionnaire in App Store Connect. Deadline: January 31, 2026.
  - **WHY**: Submission blocked without valid questionnaire after deadline.
  - **WHERE**: App Store Connect → Your App → Age Rating.

- [ ] **In-app account deletion** — Confirm the current AccountDeletionService.swift / AccountDeactivationView.swift covers: Auth deletion + Firestore data + media + Algolia + Pinecone entries + Stripe customer (if applicable) + signs user out. Must be reachable from inside the app (Settings → Account → Delete Account).
  - **WHY**: Mandatory per App Store Review Guideline 5.1.1(v) since 2022.
  - **WHERE**: Test the full deletion flow on a real account. Agent implemented it; human must verify the end-to-end path is reachable and complete.

- [ ] **UGC Moderation process (24-hour SLA)** — Apple requires responses within 24 hours: remove content AND eject the offending user within one day. This requires a human-watched queue.
  - **WHY**: Guideline 1.2 — a "report" button that files into a void is not compliant; you need an actual moderation workflow.
  - **WHERE**: Set up a dashboard/inbox (Firebase console, email alias, or moderation tool like Hive/Perspective) that you or a delegate actively watches daily. (HYBRID: agent adds Report/Block/Mute UI + Cloud Functions; human sets up the workflow.)

- [ ] **EULA / Terms / Privacy Policy content** — Content must explicitly prohibit objectionable content and abusive behavior; Privacy Policy must describe all data collected. Recommend legal review.
  - **WHY**: Required by Guideline 1.2 for UGC apps; GDPR/CCPA compliance.
  - **WHERE**: Work with a lawyer; host on your domain; agent wires the signup gate + in-app links (HYBRID).

- [ ] **APNs PRODUCTION cert/key** — Upload the production APNs key (not dev) to Firebase Cloud Messaging.
  - **WHY**: Dev push notifications work on dev builds; production builds require the production APNs key. Silent failure in production.
  - **WHERE**: Firebase Console → Project Settings → Cloud Messaging → iOS App Configuration.
