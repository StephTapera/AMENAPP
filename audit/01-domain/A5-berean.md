# AUDIT A5: BEREAN SURFACE
## Comprehensive AI Safety & Citation Audit

**Auditor:** Agent A5  
**Date:** 2026-06-07  
**Surface:** Berean — scripture study, prayer, reflection, church notes, life guidance, live voice  
**Strict Requirements:**
- ALWAYS-CITE: every AI response must cite its source
- MULTI-PERSPECTIVE: contested theology must present multiple views
- CRISIS SIGNAL → REAL HUMAN: no AI-only crisis responses
- FAIL-CLOSED: refuse > fabricate
- CLAUDE-ONLY: no fallover to other models

## CRITICAL P0 FINDINGS

### A5-001: AI_ROUTE_VIOLATION — Hardcoded OpenAI Fallover for Live Voice

**SEVERITY:** P0  
**SURFACE:** BereanLiveVoiceView → BereanVoiceSessionManager → BereanRealtimeWebSocketTransport  
**TYPE:** AI_ROUTE_VIOLATION  
**EVIDENCE:** `/AMENAPP/AIIntelligence/BereanRealtimeWebSocketTransport.swift:113`

```swift
let resolvedModel = model ?? clientSecret.model ?? "gpt-realtime"
guard let encodedModel = resolvedModel.addingPercentEncoding(withAllowedCharacters: .urlQueryAllowed),
      let url = URL(string: "wss://api.openai.com/v1/realtime?model=\(encodedModel)") else {
```

**EXPECTED:** Berean MUST use Claude exclusively with retry+backoff then graceful error. NO fallover to other models.

**ACTUAL:** Live voice sessions connect to OpenAI's realtime API (`wss://api.openai.com/v1/realtime`). When `model` is nil, defaults to `"gpt-realtime"` (GPT-4 realtime). Client secret model can also be OpenAI-issued.

**IMPACT:** Users conducting live voice prayer/reflection/scripture study are processed by GPT, not Claude. Violates the CLAUDE-ONLY requirement for Berean.

**FIX_PATH:**
1. Broker (`createRealtimeSession` Cloud Function) MUST issue only Claude-compatible secrets
2. Fallover logic must use Claude Realtime (not OpenAI), or gracefully degrade to turn-based + retry
3. Remove hardcoded `"gpt-realtime"` default; throw error if model is missing
4. Add test: verify model is Claude-family before connecting

**HUMAN_GATE:** yes

### A5-002: SAFETY_GAP — Crisis Detection Silently Fails Open

**SEVERITY:** P0  
**SURFACE:** BereanStudyService (prayer/reflection), Selah (crisis routes), CrisisDetectionService  
**TYPE:** SAFETY_GAP  
**EVIDENCE:** `/AMENAPP/CrisisDetectionService.swift:298-305`

```swift
} catch {
    dlog("❌ [CRISIS] AI API error: \(error)")
    
    // Fallback: No crisis detected if AI fails
    return CrisisDetectionResult(
        isCrisis: false,
        crisisTypes: [],
        urgencyLevel: .none,
        recommendedResources: [],
        confidence: 0.0,
        suggestedIntervention: .none
    )
}
```

**EXPECTED:** If crisis AI analysis fails, return `isCrisis = true` (fail-closed). Route to real human or crisis hotline.

**ACTUAL:** When Firebase AI Logic Cloud Function fails (timeout, error, network issue), the service returns `isCrisis: false`. A user expressing suicidal ideation offline will receive NO crisis routing.

**IMPACT:** P0 safety gap. User in crisis during network outage or API failure will not be routed to 988 Suicide & Crisis Lifeline. This is a MISSION_VIOLATION.

**FIX_PATH:**
1. On `callFirebaseAICrisisDetectionAPI` exception, return `isCrisis: true` + critical urgency
2. Log incident with userId for immediate human follow-up
3. Trigger emergency contact escalation path
4. Fallback: always show crisis resources (never hide them)

**HUMAN_GATE:** yes

### A5-003: CITATION_MISSING — Daily Verse Has No Source Attribution

**SEVERITY:** P1  
**SURFACE:** AIDailyVerseView (home feed, lock screen)  
**TYPE:** MISSION_VIOLATION  
**EVIDENCE:** `/AMENAPP/AIDailyVerseView.swift:116-117`, `/AMENAPP/DailyVerseGenkitService.swift:156-173`

```swift
// AIDailyVerseView.swift, line 116:
SelahView(
    message: bereanMessage(from: verse),
    originalQuery: "Daily Verse"
)

// bereanMessage(from:) on line 251:
BereanMessage(
    id: UUID(),
    content: "\(verse.text)\n\n— \(verse.reference)",
    ...
)
```

The Genkit Cloud Function `generateDailyVerse` returns `verse["reflection"]` and `verse["actionPrompt"]` (AI-generated) with NO attribution stating "Claude generated this reflection" or "This action prompt is AI-drafted."

**EXPECTED:** Every AI-generated reflection, prayer prompt, and action suggestion must cite "Claude" or explicitly state "AI-generated."

**ACTUAL:** 
- Verse reference is cited ("Romans 8:28")
- Reflection is presented without any indication it was AI-generated by Claude
- Prayer prompt is not attributed
- Related verses are bare references, not sourced

**IMPACT:** P1 mission violation. Users may assume reflections are from scripture or pastoral guidance, not AI. ALWAYS-CITE is not met.

**FIX_PATH:**
1. Modify `PersonalizedDailyVerse` to include `citationSource: String` field
2. Cloud Function MUST return `"citationSource": "Claude via Genkit"` or similar
3. Update `DailyVerseBannerView` to display: "Verse: Romans 8:28 (ESV) • Reflection: AI-generated with Claude"
4. Update fallback verses to include `citationSource: "Curated"` label

**HUMAN_GATE:** no

### A5-004: MULTI_PERSPECTIVE_MISSING — Berean Answer Engine Does Not Present Contested Views

**SEVERITY:** P1  
**SURFACE:** BereanChatView, BereanAnswerEngine  
**TYPE:** RULE_HOLE  
**EVIDENCE:** `/AMENAPP/BereanAnswerEngine.swift` (examined), no multi-perspective logic found  
**EXPECTED:** Contested theology (predestination, atonement models, charismatic gifts, etc.) must present multiple denominational/theological views, not a single perspective.

**ACTUAL:** BereanAnswerEngine routes to a single Claude call with a unified system prompt. No logic branches theology by denomination or presents alternative interpretations.

**IMPACT:** P1. Users asking "Is speaking in tongues for today?" or "Is God's election compatible with free will?" receive a single theological stance, not multiple views. This violates MULTI-PERSPECTIVE.

**FIX_PATH:**
1. Add `contested_theology_topics` set (Calvinism vs. Arminianism, baptism mode, etc.)
2. When user query matches, generate TWO Claude responses:
   - "View 1: [Perspective A] (held by [traditions])"
   - "View 2: [Perspective B] (held by [traditions])"
3. Present both with equal weight and source attribution
4. Add disclaimer: "Scripture is interpreted differently by faithful Christians."

**HUMAN_GATE:** no

### A5-005: DEAD_BUTTON — "Pause" Button Does Not Pause Live Voice

**SEVERITY:** P1  
**SURFACE:** BereanLiveVoiceView  
**TYPE:** DEAD_BUTTON  
**EVIDENCE:** `/AMENAPP/BereanLiveVoiceView.swift:302-305`

```swift
Button {
    // Pause: stop the current session; a dedicated pause
    // state will be wired in a future iteration.
    Task { await vm.stopSession() }
} label: {
```

**EXPECTED:** "Pause" button pauses the session (still connected, can resume).

**ACTUAL:** "Pause" button calls `stopSession()`, which disconnects and ends the session. There is no `pauseSession()` method. The comment admits pause is not implemented.

**IMPACT:** User expects to pause briefly, but loses their session. They must restart, losing context. Minor UX bug, but labeled as "future iteration."

**FIX_PATH:**
1. Implement `BereanVoiceViewModel.pauseSession()` → calls `BereanVoiceSessionManager.pause()`
2. Add `paused` state to `BereanVoiceState` enum
3. On resume, fetch fresh token and reconnect
4. Or, simpler: rename "Pause" → "End Session" to match behavior

**HUMAN_GATE:** no

### A5-007: ORPHAN_ROUTE — BereanStudyService Does Not Route Crisis Detection

**SEVERITY:** P1  
**SURFACE:** BereanStudyService (explainVerse, prayerFromPassage, etc.)  
**TYPE:** ORPHAN_ROUTE  
**EVIDENCE:** `/AMENAPP/AIIntelligence/BereanStudyService.swift` (full file, lines 1-126), no crisis detection call  
**EXPECTED:** Study actions that process user input (e.g., draft prayer from a passage about suffering) should check for crisis signals first.

**ACTUAL:** BereanStudyService calls Firebase callables directly without running through CrisisDetectionService. If a user enters crisis content in `prayerFromPassage()`, no crisis routing occurs.

**IMPACT:** P1. User types "I want to pray about ending my pain" in the prayer drafting flow. The Cloud Function may notice but the client-side safety gate is missing.

**FIX_PATH:**
1. Before calling any Cloud Function, run: `let crisisResult = await CrisisDetectionService.shared.detectCrisis(in: userInput, userId: uid)`
2. If `crisisResult.isCrisis` and urgency is critical, show crisis card and do NOT call the study function
3. Add test case: user input with crisis language → expect crisis card, no draft generated

**HUMAN_GATE:** yes

### A5-008: CONTRACT_DRIFT — Daily Verse "Personalized" Flag Does Not Reflect Actual Personalization

**SEVERITY:** P1  
**SURFACE:** DailyVerseGenkitService  
**TYPE:** CONTRACT_DRIFT  
**EVIDENCE:** `/AMENAPP/DailyVerseGenkitService.swift:170-171, 191`

```swift
self.isPersonalized = true   // OFFLINE FIX: mark as AI-generated
...
self.isPersonalized = false  // OFFLINE FIX: fallback is not AI-personalised; retry on reconnect
```

**EXPECTED:** `isPersonalized = true` means "this verse was chosen based on the user's interests, prayer history, and liturgical context."

**ACTUAL:** The flag is set to `true` whenever Cloud Function succeeds, even if the function returned a generic fallback. The flag does not distinguish between truly personalized (user context + liturgical season) and generic-but-fresh-from-AI.

**IMPACT:** P1 contract drift. Code consumers cannot reliably know if `todayVerse` is personalized. Comment reveals intent to retry on reconnect, but the flag is not updated when that happens.

**FIX_PATH:**
1. Split into two flags: `isFromAI: Bool` and `isPersonalizedToUser: Bool`
2. Cloud Function should return `"isPersonalizedToUser": true` only if user context was used
3. On reconnect, call again and update flags accordingly
4. UI can show "Personalized for you" vs. "AI-generated" appropriately

**HUMAN_GATE:** no

**SEVERITY:** P2  
**SURFACE:** BereanRealtimeWebSocketTransport  
**TYPE:** KEY_LEAK  
**EVIDENCE:** `/AMENAPP/AIIntelligence/BereanRealtimeWebSocketTransport.swift:112-117`

```swift
request.setValue("Bearer \(clientSecret.value)", forHTTPHeaderField: "Authorization")
request.setValue("realtime=v1", forHTTPHeaderField: "OpenAI-Beta")
```

If `clientSecret.value` is an OpenAI API key (because broker issued it), the key is sent in URLRequest header directly. Device logs may capture it.

**EXPECTED:** Only Claude-compatible secrets should be issued. API keys should be server-managed.

**ACTUAL:** OpenAI keys are client-side if fallover occurs. This is a secondary concern (post-use fallover); primary concern is the fallover itself (A5-001).

**IMPACT:** P2 (mitigated by stopping fallover). If OpenAI fallover is eliminated, this is moot.

**FIX_PATH:** Same as A5-001. Once only Claude is issued, this is resolved.

### A5-010: DESIGN_VIOLATION — No Graceful Degradation for Realtime Failure

**SEVERITY:** P2  
**SURFACE:** BereanVoiceViewModel, BereanVoiceSessionManager  
**TYPE:** DESIGN_VIOLATION  
**EVIDENCE:** `/AMENAPP/AIIntelligence/BereanVoiceSessionManager.swift:24-45`

When `startAssistantSession()` fails (e.g., broker returns no model), exception is thrown and not caught by caller.

**EXPECTED:** If realtime connection cannot be established, fall back to turn-based voice input + Claude.

**ACTUAL:** Session start fails hard. User sees error or blank screen. No fallback mode.

**IMPACT:** P2. User loses prayer/study session during connectivity issue or broker outage.

**FIX_PATH:**
1. Add try/catch in BereanVoiceViewModel.startSession()
2. On failure, offer: "Realtime not available. Use text input instead? [Yes] [Retry]"
3. Transition to BereanChatView with voice-to-text mode
4. User can still pray, but without live streaming

**HUMAN_GATE:** no

## LOW P3 FINDINGS

### A5-011: MISSING_STATE — Voice Transcript Not Sanitized for Prompt Injection

**SEVERITY:** P3  
**SURFACE:** BereanRealtimeWebSocketTransport → persistRealtimeTranscriptChunk  
**TYPE:** MISSING_FEATURE  
**EVIDENCE:** `/AMENAPP/AIIntelligence/BereanRealtimeWebSocketTransport.swift:77-92`

```swift
func persistTranscriptChunk(
    sessionId: String,
    text: String,
    ...
```

The `text` (from speech-to-text) is not sanitized before persisting. A user with a malicious prompt injection in their prayer/reflection could theoretically alter backend behavior.

**EXPECTED:** Use `BereanContextCoordinator.sanitizeCommunityContent(text)` before persisting.

**ACTUAL:** Transcript is stored as-is.

**IMPACT:** P3 (low): Speech-to-text is user's own words, unlikely to be adversarial. But if user's device is compromised, injection is possible.

**FIX_PATH:** Wrap text: `let sanitized = BereanContextCoordinator.sanitizeCommunityContent(text)` before persistence.

**HUMAN_GATE:** no

### A5-012: DESIGN_VIOLATION — Multiple Realtime Transports Can Exist

**SEVERITY:** P3  
**SURFACE:** BereanVoiceSessionManager  
**TYPE:** DESIGN_VIOLATION  
**EVIDENCE:** `/AMENAPP/AIIntelligence/BereanVoiceSessionManager.swift:10-12`

```swift
private let sessionManager: BereanRealtimeSessionManager
private let analyticsService: BereanRealtimeAnalyticsService
private let transport: BereanRealtimeWebSocketTransport
```

Each BereanVoiceSessionManager instance creates its own transport. If a user starts two voice sessions simultaneously (e.g., opens BereanLiveVoiceView twice), two transports connect to realtime broker.

**EXPECTED:** Singleton or limited-pool transport; only one active session per user.

**ACTUAL:** Transport is not a singleton; caller can create multiple managers.

**IMPACT:** P3. Low impact (UI typically prevents this). But two simultaneous realtime sessions consume tokens and cost money.

**FIX_PATH:** Make BereanVoiceSessionManager.shared a singleton, or validate only one session per user.

**HUMAN_GATE:** no
