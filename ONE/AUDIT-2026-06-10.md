## CONFIRMED — Functional defect

### H-1 · Relay button does not enforce `forwardAllowed` (sticky-consent bypass) — HIGH
- **Where:** `AMENAPP/AMENAPP/ONE/World/ONEWorldFeedView.swift:332` (button gate), `:336` (action), `:267` (cosmetic "No forwarding" badge); service `AMENAPP/AMENAPP/ONE/World/ONEFeedModeService.swift:81`; uncalled enforcement primitives in `AMENAPP/AMENAPP/ONE/Differentiators/ONEStickyConsentService.swift:33,48`.
- **Invariant:** SECURITY.md §3.1 ("per-message permissions enforced in UI"); CONTRACTS.md §5; `ONEStickyConsentService` header rule "a `forwardAllowed=false` moment cannot be relayed."
- **What's wrong:** The Relay button is gated only on `reachBudget.hasReachRemaining && userRelayBudget > 0`. It never checks `item.permissions.forwardAllowed`, and `relay(itemID:)` calls `one_relayMoment` with no consent check. A forward-prohibited moment renders the "No forwarding" badge **and** an active Relay button simultaneously — the UI contradicts its own displayed state. `ONEStickyConsentService.isPermitted(.forward,…)` exists but has zero call sites on the relay path. The server-side CF reject is P5-deferred, so today there is **no** enforcement layer at all — exactly where the contract claims client-side enforcement is live.
- **Fix:** Gate the Relay button (and `ONEFeedModeService.relay`) on `ONEStickyConsentService.isPermitted(.forward, for: item)`; disable/hide the button when `!forwardAllowed`.

## UNCERTAIN — Defensive fix recommended

### U-1 · `< 0.7 → .unknown` enforced only in derived `displayClassification`, not the stored field — (claimed HIGH; lenses split)
`AMENAPP/AMENAPP/ONE/Core/ONEProvenanceModels.swift:17`. The stored `classification` is Codable-encoded raw; only the computed display property degrades. `ONEProvenanceLabelService.swift:81` actually emits `(.captured, 0.55)`. **All three shipped UI consumers correctly call `displayClassification`**, so nothing user-facing fakes certainty today — hence the split. Exposure is to Firestore persistence and any future non-UI / Cloud-Function reader of `.classification`. The contract-text lens noted those CF readers are themselves P5-deferred. **Recommendation:** normalize at the model boundary (init/encoder) so confidence `< 0.7` forces `.unknown` *before* persistence — single source of truth, cheap, removes the foot-gun.
